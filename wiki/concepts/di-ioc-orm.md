# IoC, DI, ORM

## IoC (Inversion of Control - 제어의 역전)

객체 지향 프로그래밍에서 객체의 생성과 실행의 주도권을 개발자가 아닌 프레임워크가 담당하게 하는 디자인 원칙. 기존에는 객체가 스스로 의존성을 관리하거나 생성했지만, IoC를 적용하면 객체의 제어를 외부(프레임워크 또는 컨테이너)에 위임하여 코드의 유연성과 재사용성을 높일 수 있습니다.

기존 흐름에서는 자신이 필요한 객체(의존성)를 스스로 생성하거나 관리했지만, IoC를 통해 객체가 필요로 하는 의존성을 외부에서 주입받게 됩니다. 따라서 객체 간 결합도를 낮추어 코드가 더 유연하고 테스트하기 쉬워집니다.

## DI (Dependency Injection - 의존성 주입)

IoC를 구현하는 방법 중 하나로, 객체가 동작하는 데 필요한 의존성을 외부에서 주입하는 디자인 패턴입니다.

**주입방식:**
- Constructor Injection: 생성자를 통해 의존성을 주입
- Setter Injection: setter 메서드를 이용하여 주입
- Interface Injection: 인터페이스를 활용한 주입

이로 인해 객체 간의 강한 결합이 줄어들고, 객체는 자신의 역할만 수행하고 의존성을 외부에서 주입받으면서 재사용성과 테스트의 용이성이 증가합니다.

## IoC와 DI의 관계

IoC는 제어의 주체를 바꿈으로써 프로그램이 동작하는 방식을 변화시키는 큰 개념이고, DI는 이를 구체적으로 구현하기 위한 방법 중 하나입니다.

## ORM (Object-Relational Mapping)

데이터베이스의 데이터를 객체(Object) 형태로 매핑하여 객체 지향적으로 데이터를 조작할 수 있도록 하는 프로그래밍 기법. SQL을 직접 작성하지 않고도 데이터베이스와 상호작용할 수 있게 해줍니다.

**특징과 원리:**
- 데이터베이스 테이블과 클래스가 1:1로 매핑됩니다. 테이블의 행(Row)은 객체(Object)로 매핑됩니다.
- 직접 SQL 작성 불필요: 일반적으로 SQL 쿼리 대신 ORM 프레임워크가 제공하는 메서드를 호출하면 SQL이 자동으로 생성되고 실행됩니다.

**장점:**
- 생산성 증가: SQL 작성 없이 객체를 통해 데이터를 처리할 수 있어 개발 속도가 빨라집니다.
- 유지보수 용이성: 객체 지향 방식으로 코드를 작성하므로 코드 가독성과 유지보수가 더 쉽습니다.
- DBMS 독립성: ORM 라이브러리를 통해 데이터베이스 변경 시 코드 변경을 최소화할 수 있습니다.

**단점:**
- 학습 곡선이 다소 높고, 복잡한 쿼리를 작성하기 어려울 수 있다.
- 성능이 중요한 경우 직접 SQL 작성보다 상대적으로 느릴 수 있다.
