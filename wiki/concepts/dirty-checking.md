# Dirty Checking이란 무엇인가요?

**정의:** 객체의 속성이 변경되었는지를 검토하여 데이터베이스에 업데이트할 필요성을 판단하는 과정입니다. 변경된 데이터만 데이터베이스에 반영하여 불필요한 작업을 줄입니다.

## 동작 원리

- **상태 추적**: 영속성 컨텍스트가 관리하는 객체의 원래 상태와 현재 상태를 비교합니다. 객체가 로드될 때 초기 상태를 저장하고, 이후 변경된 속성을 추적합니다.
- **변경 감지**: 객체의 속성이 변경될 때마다 이를 감지하며, 트랜잭션이 완료되기 전에 변경사항을 데이터베이스에 반영합니다.

## 구현

- **가변 객체**: 객체의 프로퍼티가 변경될 때마다, 관련된 데이터 구조나 상태를 업데이트합니다. 이를 위해 getter와 setter 메서드를 활용합니다.
- **비교 방법**: 객체의 해시코드나 이퀄스 메서드를 사용하여 두 상태를 비교하거나, 각 속성을 개별적으로 비교하는 방법이 있습니다.

## 장단점

**장점:**
- 데이터베이스와 애플리케이션 간의 통신을 최소화하여 성능을 향상시킵니다.
- 명시적으로 SQL 쿼리를 작성할 필요가 없어 코드의 간결함을 증가시킵니다.

**단점:**
- 복잡한 객체 그래프에서는 상태 비교가 비효율적일 수 있습니다.
- 대규모 객체 상태 추적 시 메모리 사용량이 증가할 수 있습니다.

## 비유를 통한 설명 - 음식점과 주방 재료 관리

1. **음식점과 요리사**: 음식점은 주방과 손님들, 즉 영속성 컨텍스트와 클라이언트를 비유합니다. 요리사는 객체와 그 상태를 추적하는 역할을 합니다.
2. **재료 상태 관리**: 요리사가 요리를 시작할 때, 주방에는 신선한 재료들이 준비되어 있습니다. 이 상태가 객체의 초기 상태입니다. 요리 중에 재료가 소모되거나 변경되면(토마토 추가, 양파 양 조절), 요리사는 주방의 재료 목록을 확인하고 변경된 사항을 관리해야 합니다.
3. **Dirty Checking 동작**: 요리사는 요리를 진행하면서 주방의 재료 목록을 체크합니다. 이 과정이 Dirty Checking입니다. 새롭게 조리된 요리와 새로 추가된 재료를 기억하고, 어떤 것들이 변경되었는지를 확인합니다.
4. **음식 준비 완료**: 요리가 완성되면 요리사는 변경된 재료 목록을 기반으로 주문서에 반영합니다.
5. **전체적인 효율성**: 주방에선 대규모 재료 목록을 항상 점검할 필요가 없고, 요리사가 준비한 요리에서만 변경된 사항을 따로 체크하여 메뉴를 업데이트합니다. 이는 성능을 최적화하고, 불필요한 작업을 줄이는 효과가 있습니다.

**요약:** 불필요한 재료 목록 전체를 매번 확인하는 것이 아니라, 필요한 부분만 체크하여 효율적으로 작업하는 것이 Dirty Checking의 핵심입니다.
