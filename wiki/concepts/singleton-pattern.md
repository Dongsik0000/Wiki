# 싱글턴 패턴

**정의:** 클래스의 인스턴스를 오직 하나만 생성하도록 보장하는 디자인 패턴. 전역 접근점을 제공하여 해당 인스턴스를 어디서나 접근할 수 있게 함.

## 핵심 특징

- 단 하나의 인스턴스만 생성 가능
- 전역 접근 지점 제공
- 최초 한 번만 메모리 할당
- 이후 동일한 인스턴스 재사용

## 장점

- 자원 관리: 애플리케이션이 실행되는 동안 하나의 인스턴스만 유지되므로 메모리 및 자원을 효율적으로 관리할 수 있습니다.
- 통합된 상태 관리: 공통된 상태를 관리하거나 설정을 공유해야 할 경우 유용합니다.
- 간결한 인터페이스 제공: 인스턴스에 대한 접근을 간단하게 할 수 있습니다.

## 단점

- 글로벌 상태 유지로 인한 문제: 전역 상태를 가진 인스턴스를 사용하면 의존성이 높아져서 코드의 유지보수가 어려워질 수 있습니다.
- 테스트의 어려움: 싱글턴 클래스를 테스트하기가 어려워질 수 있으므로, mocking이나 의존성 주입을 통해 테스트를 어렵게 만들 수 있습니다.
- 스레드 안전성 문제: 멀티스레드 환경에서 동기화 문제로 인해 올바르게 구현되지 않을 수 있습니다.

## 일반적인 구현 방법

```java
public class Singleton {
    private static Singleton instance;

    // private 생성자
    private Singleton() {}

    // 공용 접근 메서드
    public static Singleton getInstance() {
        if (instance == null) {
            instance = new Singleton();
        }
        return instance;
    }
}
```

## 비유를 통한 이해

1. **단 하나의 인스턴스**: 나라에는 오직 하나의 대통령만 있습니다. 여러 명의 대통령이 존재할 수는 없습니다.
2. **전국민의 접근**: 모든 국민은 대통령에게 연락할 수 있지만, 모두가 직접 대통령의 집에 가서 만나지는 않습니다. 대신 대통령을 호출할 수 있는 방법이 있습니다.
3. **대통령의 권한**: 대통령이 중요한 결정을 내릴 때, 다른 부서가 이를 곧바로 반영해야 합니다. 모든 부분에서 동일한 인스턴스를 참조하므로 상태가 일관되게 유지됩니다.
4. **정보 공유**: 대통령이 중요한 국가 전략을 세운다면, 이 정보는 국가의 모든 정책에 영향을 미칠 수 있습니다.
5. **멀티스레드 환경에서의 도전**: 여러 국가가 동시에 대통령을 선출하려 한다면 혼란이 발생할 것입니다. 싱글턴 패턴에서도 멀티스레드 환경에서 인스턴스를 안전하게 생성해야 하는 것이 필요합니다.
