# LLM Wiki 설계 문서

- 작성일: 2026-07-16
- 상태: 승인됨

## 배경

RAG 기반 문서 질의응답과 달리, LLM이 소스를 읽을 때마다 지식을 처음부터 다시 조립하는 대신, 소스를 하나씩 통합(ingest)하면서 누적되는 마크다운 위키를 LLM이 직접 쓰고 유지보수하는 개인 지식 베이스를 만든다. 사람은 소스 선정과 질문을, LLM은 요약·상호참조·정리를 담당한다.

## 목적 및 범위

- **용도**: 개인 전체 개발 지식 위키. 특정 프로젝트 하나에 국한되지 않고, 여러 프로젝트와 기술 학습을 아우른다.
- **주로 담을 내용**: 기술 학습 자료, 개발 결정/설계 기록, 프로젝트 진행 로그.
- **언어**: 위키 문서(요약, 개념 페이지 등)는 한국어로 작성한다.
- **Ingest 방식**: 소스를 한 번에 하나씩, 사용자와 대화하며 통합한다 (배치 처리 아님).
- **Obsidian**: 처음 사용해보며, Claude + Obsidian 워크플로우(에이전트가 위키를 쓰고, Obsidian으로 그래프 뷰·상호링크를 확인)가 이 프로젝트의 핵심 목적이다. winget으로 설치하고, 이 Wiki 폴더를 vault로 사용한다.
- **버전 관리**: 로컬 git 저장소로 관리한다. (2026-07-16 업데이트: GitHub 원격

## 아키텍처: 3계층 구조

원본 LLM Wiki 패턴 문서가 설명하는 3계층을 그대로 따른다.

1. **raw/** — 원본 소스. 불변(immutable). LLM은 읽기만 하고 절대 수정하지 않는다.
2. **wiki/** — LLM이 생성·관리하는 마크다운 지식 페이지. 사용자는 읽고, LLM이 쓴다.
3. **CLAUDE.md** — 스키마 문서. 위키 구조, 컨벤션, ingest/query/lint 워크플로우를 LLM에게 지시한다.

## 디렉토리 구조

```
Wiki/
├── CLAUDE.md
├── raw/
│   ├── articles/      # 학습 자료 원문
│   ├── decisions/     # 개발 결정을 논의한 원본 메모
│   ├── logs/          # 프로젝트 진행 관련 원본 기록
│   └── assets/        # 이미지 등 첨부파일
└── wiki/
    ├── index.md       # 전체 카탈로그
    ├── log.md         # 시간순 작업 이력
    ├── projects/      # 프로젝트별 개요 페이지
    ├── concepts/      # 기술 개념 페이지 (프로젝트 간 재사용)
    └── decisions/     # 설계/아키텍처 결정 기록 페이지
```

## 페이지 규격

### Frontmatter

```yaml
---
title: 페이지 제목
type: project | concept | decision | source-summary
tags: [태그1, 태그2]
created: YYYY-MM-DD
updated: YYYY-MM-DD
related: ["[[다른-페이지]]"]
---
```

### index.md

카테고리(프로젝트/개념/결정)별로 다음 형식의 목록을 유지한다:

```
- [[페이지명]] — 한줄 요약 (업데이트일)
```

소스를 ingest하거나 새 페이지를 만들 때마다 즉시 갱신한다.

### log.md

Append-only. 각 항목은 일관된 접두사로 시작해 `grep`으로 파싱 가능하게 한다:

```
## [YYYY-MM-DD] ingest | 소스 제목
## [YYYY-MM-DD] query | 질문 요약
## [YYYY-MM-DD] lint | 점검 결과 요약
```

## 워크플로우

### Ingest

1. 사용자가 raw/에 소스를 추가하고 처리를 요청.
2. LLM이 소스를 읽고 핵심 내용을 사용자와 논의.
3. 요약 페이지(source-summary) 또는 관련 project/concept/decision 페이지를 생성·갱신.
4. index.md, log.md 갱신.

### Query

1. 사용자가 질문.
2. LLM이 index.md에서 관련 페이지를 찾아 읽고 답변을 종합 (출처 인용).
3. 답변이 재사용 가치가 있으면 새 위키 페이지로 파일링할지 제안.

### Lint

주기적으로 실행. 점검 항목:
- 페이지 간 모순
- 새 소스로 인해 낡은 주장
- 고아 페이지 (들어오는 링크 없음)
- 언급만 되고 페이지가 없는 개념
- 누락된 상호참조

## CLAUDE.md 스키마에 명시할 핵심 규칙

- raw/는 절대 수정하지 않는다. 편집 대상은 wiki/ 뿐이다.
- 위키 문서는 한국어로 작성한다.
- **에이전트 원칙 예외 명시**: 사용자의 전역 CLAUDE.md에는 "코드 변경 전 파일/줄 번호와 함께 복사-붙여넣기용으로 제시하고 에이전트가 직접 수정하지 않는다"는 규칙이 있다. 이 규칙은 애플리케이션 소스 코드(예: 검색 스크립트 등 실행 코드)에 적용되는 것으로, 위키를 구성하는 마크다운 문서(요약, 개념 페이지, index.md, log.md 등)에는 적용되지 않는다. 위키 마크다운은 LLM이 직접 생성·수정한다. 단, 이 위키 프로젝트 내에서 실행 가능한 코드(예: 검색 스크립트)를 작성하게 될 경우에는 전역 규칙을 따른다.

## Obsidian 설정

- winget (`Obsidian.Obsidian`)으로 설치 완료.
- Wiki 폴더(`raw/`, `wiki/`가 있는 이 디렉토리) 자체를 Obsidian vault로 연다 — 별도 vault 폴더를 만들지 않는다.
- 첨부파일(이미지) 저장 경로는 `raw/assets/`로 지정한다 (Settings → Files and links → Attachment folder path). 이 설정은 vault를 처음 연 뒤 GUI에서 수동으로 지정해야 한다 (앱을 실행하기 전에는 `.obsidian` 설정 폴더가 생성되지 않음).
- "Download attachments for current file" 단축키 바인딩은 사용자가 Settings → Hotkeys에서 직접 검색해 지정한다 (원본 패턴 문서의 팁).
- 그래프 뷰, 위키링크(`[[페이지명]]`) 탐색은 Obsidian GUI에서 바로 확인 가능하도록 wiki/ 페이지들을 상호링크 형식으로 작성한다.

## 범위 제외 (이번 단계)

- 검색 도구(qmd 등) 구축은 하지 않는다. 위키가 작고 index.md만으로 충분한 초기 단계이기 때문.
- 초기 raw source ingest는 하지 않는다. 이번에는 빈 구조(뼈대)만 만든다.
